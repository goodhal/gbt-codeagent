## 6 源代码漏洞测试内容

### 6.1 源代码漏洞分类

本标准根据软件开发中常用的概念来组织 C/C++ 语言源代码漏洞的类别，具体分为如下 **8 个类别**：

| 编号 | 类别 | 说明 |
|------|------|------|
| a) | 行为问题 | 由于应用程序的意外行为而引发的漏洞 |
| b) | 路径错误 | 不恰当的处理访问路径而引发的漏洞 |
| c) | 数据处理 | 在处理数据的功能中发现的问题 |
| d) | 错误的 API 协议实现 | 由于软件未按预期使用 API 而引起的漏洞 |
| e) | 不充分的封装 | 由于缺少封装敏感数据或功能而引起的漏洞 |
| f) | 安全功能 | 软件安全功能如身份鉴别、访问控制、机密性、密码学和特权管理等相关的漏洞 |
| g) | Web 问题 | Web 技术相关的漏洞 |
| h) | 用户界面错误 | 与用户界面相关的漏洞 |

> 注：C/C++ 语言源代码漏洞的名称及其所属的类别参见附录 B。

## 6.2.1 行为问题

### 问题分类 GB/T34943-6.2.1.1 不可控的内存分配 🟠

**严重级别**: HIGH | **CWE**: [CWE-770](https://cwe.mitre.org/data/definitions/770.html)

内存分配的大小受外部控制的输入数据影响，且程序没有指定内存分配大小的上限。

**漏洞描述**:
攻击者可以使程序分配大量的内存，程序可能会因为内存资源不足而崩溃。

**修复或规避建议**:
在程序中指定内存分配大小的上限，在分配内存前对要分配的内存大小进行验证，确保要分配的内存大小不超过上限。

**审计要点**:
检查是否存在无上限的内存分配调用（`malloc(userInput * sizeof(char))`、`new String[length]`），参数是否来自用户输入且未做边界检查。

## 6.2.2 路径错误

### 问题分类 GB/T34943-6.2.2.1 不可信的搜索路径 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-426](https://cwe.mitre.org/data/definitions/426.html)

程序使用关键资源时没有指定资源的路径，而是依赖操作系统去搜索资源。

**漏洞描述**:
攻击者可以在搜索优先级更高的文件夹中放入相同名称的资源，程序会使用攻击者控制的资源。

**修复或规避建议**:
使用关键资源时指定资源所在的路径。

**审计要点**:
检查 `system(command)` 中 PATH 是否被依赖；`LoadLibrary(path)` 的路径参数是否来自用户输入；是否存在 DLL/SO 劫持风险。

## 6.2.3 数据处理

### 问题分类 GB/T34943-6.2.3.1 相对路径遍历 🟠

**严重级别**: HIGH | **CWE**: [CWE-22](https://cwe.mitre.org/data/definitions/22.html)

路径名受外部控制的输入数据影响，且程序没有使能够解析到目录外位置的字符序列（如"../"）失效。

**漏洞描述**:
攻击者可以通过输入能够解析到目录外位置的字符序列来访问限制目录之外的文件或目录。

**修复或规避建议**:
在构建路径名前对输入数据进行验证，确保外部输入仅包含允许的构成路径名的字符。

**审计要点**:
检查文件操作函数（`fopen/open/access`）的路径参数是否直接拼接用户输入（如 `strcat(path, filename)`）；是否存在 `../` 路径遍历风险；是否使用了白名单校验文件名。

---

### 问题分类 GB/T34943-6.2.3.2 绝对路径遍历 🟠

**严重级别**: HIGH | **CWE**: [CWE-36](https://cwe.mitre.org/data/definitions/36.html)

路径名由外部控制的输入数据决定，且程序没有限制路径名允许访问的目录。

**漏洞描述**:
攻击者可以通过输入路径名来访问任意的文件或目录。

**修复或规避建议**:
在程序中指定允许访问的文件或目录，在访问文件或目录前对路径名进行验证，确保仅允许访问指定的文件或目录。

**审计要点**:
检查 `fopen(absolutePath, "r")` 等直接使用用户输入作为绝对路径的操作；是否有白名单限制可访问的目录。

### 问题分类 GB/T34943-6.2.3.3 命令注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-78](https://cwe.mitre.org/data/definitions/78.html)

使用未经验证的输入数据构建命令。

**漏洞描述**:
攻击者可执行任意命令（包括 delete 等恶意命令）。

**修复或规避建议**:
在构建命令前对输入数据进行验证，确保输入数据仅能用于构成允许的命令。

**审计要点**:
- `system(command)` / `popen(cmd, "r")` 的 command 参数是否来自用户输入
- 是否存在 `strcat(cmd, "\\"); strcat(cmd, command); system(cmd);` 等危险拼接
- 是否有分号/管道符注入风险
- 是否使用白名单校验命令和参数

---

### 问题分类 GB/T34943-6.2.3.4 SQL 注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-89](https://cwe.mitre.org/data/definitions/89.html)

使用未经验证的输入数据采用拼接字符串的方式形成 SQL 语句。

**漏洞描述**:
攻击者可植入任何 SQL 语句，实现越权查询数据库的敏感数据、非法修改数据库数据和提升攻击者的数据权限等目的。

**修复或规避建议**:
在拼接 SQL 语句前对输入数据进行验证，确保输入数据不包含 SQL 语句的关键字符；或者采用参数化 SQL 查询语句，并将输入数据作为 SQL 语句的参数。

**审计要点**:
- `strcat(sqlQuery, name);` 是否拼接用户输入到 SQL 语句
- 是否使用参数化查询替代字符串拼接
- 排序字段/动态表名是否做了白名单验证

---

### 问题分类 GB/T34943-6.2.3.5 进程控制 🟠

**严重级别**: HIGH | **CWE**: [CWE-114](https://cwe.mitre.org/data/definitions/114.html)

使用未经验证的输入数据作为动态加载库的标识符。

**漏洞描述**:
使用未经验证的输入数据作为动态加载库的标识符将使攻击者有机会加载恶意的代码库。

**修复或规避建议**:
在动态加载库前对输入数据进行验证，确保输入数据仅能用于加载允许加载的代码库。

**审计要点**:
- `LoadLibrary(path)` / `dlopen(path, ...)` 的路径参数是否来自用户输入
- 是否有类名/库名白名单限制
- `HANDLE hlib = LoadLibrary(path);` 是否安全

---

### 问题分类 GB/T34943-6.2.3.6 缓冲区溢出 - 栈溢出 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-121](https://cwe.mitre.org/data/definitions/121.html)

对被分配内存空间之外的内容空间进行读或写操作。

**漏洞描述**:
攻击者可利用缓冲区溢出让系统崩溃或者执行恶意代码。

**修复或规避建议**:
在对缓冲区进行读写操作时，对该写缓冲区的数据长度进行检查，确保读出的内容在分配的空间之内。

**审计要点**:
- `scanf("%s", value);` 未限制长度
- `strcpy(password, parameter);` / `strcat(msg, PATH);` 无边界检查
- `gets(value);` 等危险函数
- 应使用 `strncpy`/`snprintf` 替代并正确设置长度限制

---

### 问题分类 GB/T34943-6.2.3.7 格式化字符串漏洞 🟠

**严重级别**: HIGH | **CWE**: [CWE-134](https://cwe.mitre.org/data/definitions/134.html)

printf 函数中的格式化字符串受外部输入数据影响，这可能会导致缓冲区溢出或数据显示异常问题。

**漏洞描述**:
攻击者可输入恶意的格式化字符串造成缓冲区溢出，进而导致系统崩溃或者执行恶意代码。

**修复或规避建议**:
确保向所有格式化字符串函数都传递一个不能由用户控制的静态格式化字符串，并且该参数紧接在正确的数量参数之后。

**审计要点**:
- `printf(s);` — s 中可能包含"%n"等格式控制字符导致缓冲区溢出
- `printf("%s", s);` — 传递一个不能由用户控制的静态格式化字符串 ✅
- `fprintf`/`sprintf`/`snprintf` 同理

---

### 问题分类 GB/T34943-6.2.3.8 整数溢出 🟠

**严重级别**: HIGH | **CWE**: [CWE-190](https://cwe.mitre.org/data/definitions/190.html)

使用未经验证的整型数据进行算术运算，可能导致计算结果过大而无法在系统位数范围内保存。

**漏洞描述**:
攻击者可输入过大的数据引发软件崩溃或破坏系统中重要内存等安全事件。

**修复或规避建议**:
在对来自用户的整型数据作算术运算前进行验证，确保运算结果不会溢出。

**审计要点**:
- `unsigned int num = 1000;` 但实际假设运行在32位系统中时 `nsrp * 4` 可能溢出
- `malloc((nsrp + num) * sizeof(char *))` 乘法溢出后分配空间不够
- size_t/int 加法/乘法运算前是否检查溢出

### 问题分类 GB/T34943-6.2.3.9 信息通过错误消息泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-209](https://cwe.mitre.org/data/definitions/209.html)

软件呈现给用户的错误消息中包括与环境、用户或相关数据有关的敏感信息。

**漏洞描述**:
对攻击者而言，敏感信息可能本身就是有价值的信息，或者有助于开展进一步的攻击。

**修复或规避建议**:
确保错误消息中仅含有对目标受众有用的少量细节。

**审计要点**:
- 错误输出是否包含完整路径 `strcat(msg, PATH); does not exist!`
- 是否输出配置目录的完整路径名
- 错误响应是否只返回通用消息 `"Sorry! We will fix the problem soon!"`

---

### 问题分类 GB/T34943-6.2.3.10 信息通过服务器日志文件泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-532](https://cwe.mitre.org/data/definitions/532.html)

将敏感信息写入服务器日志文件。

**漏洞描述**:
攻击者有机会通过访问日志文件读取敏感信息。

**修复或规避建议**:
慎重考虑写入日志文件信息的隐秘性。不要把敏感信息写入日志文件。

**审计要点**:
- `write_log()` 写入内容是否包含 username/password
- 日志记录是否包含密码/token/身份证号等敏感信息
- `if(is_trusted(username, password)) { write_log(...); }` 仅可信时才记录

---

### 问题分类 GB/T34943-6.2.3.11 信息通过调试日志文件泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-532](https://cwe.mitre.org/data/definitions/532.html)

应用程序没有充分限制对用于调式的日志文件的访问。

**漏洞描述**:
调试日志文件通常包含应用程序的敏感信息，攻击者有机会通过访问调试日志文件读取敏感信息。

**修复或规避建议**:
在产品发布之前移除产生日志文件的代码。

**审计要点**:
检查 CCLOG/AddressBookId 等调试日志是否在发布版本中移除；调试日志路径是否暴露。

---

### 问题分类 GB/T34943-6.2.3.12 未检查的输入作为循环条件 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-606](https://cwe.mitre.org/data/definitions/606.html)

软件没有对被用于循环条件的输入进行适当的检查。

**漏洞描述**:
攻击者可以让软件循环过多从而使软件拒绝服务。

**修复或规避建议**:
规定循环次数的上限，在将用户输入的数据用于循环条件前验证用户输入的数据是否超过上限。

**审计要点**:
- `if(count >0){ for(i = 0;i <count;i++){}` 循环次数由用户控制
- `if(count >0 && count < MAX_COUNT)` ✅ 有上限检查

---

### 问题分类 GB/T34943-6.2.3.13 通过用户控制的 SQL 关键字绕过授权 🟠

**严重级别**: HIGH | **CWE**: [CWE-89](https://cwe.mitre.org/data/definitions/89.html)

软件使用的数据库表其中包含了某个用户不应有权访问的记录，但该软件执行的一个 SQL 语句中的关键字段可以受用户控制。

**漏洞描述**:
如果用户可以将关键字设置为任何值，那么该用户就可以修改该关键字指向未经授权的记录，实现水平越权访问其他用户的数据。

**修复或规避建议**:
对用户输入的关键字进行验证，确保其只能访问该用户有权访问的记录。使用白名单验证动态列名和表名。

**审计要点**:
- ORDER BY 排序字段是否由用户输入控制
- 数据库表中的 staftID/departID 等关键筛选条件是否可被用户操控
- 动态表名/列名是否做白名单验证
- 是否存在通过修改查询条件访问其他用户数据的可能

---

### 问题分类 GB/T34943-6.2.3.14 未使用盐值计算散列值 🟠

**严重级别**: HIGH | **CWE**: [CWE-759](https://cwe.mitre.org/data/definitions/759.html)

软件针对口令等不应可选的输入使用了单向加密数列，但未使用盐值。

**漏洞描述**:
如果未使用盐值计算散列值，攻击者就更容易利用彩虹表等字典式攻击技术破解口令。

**修复或规避建议**:
使用盐值计算散列值，增加攻击者破解口令的难度。

**审计要点**:
- `SHA256 sha256; sha256.enc(input, output);` 直接哈希无盐值
- `send_salt(salt); AESEncryption aesEncryptor; aesEncrypto.enc(salt, output);` 正确使用盐值 ✅
- 密码存储必须加盐

## 6.2.4 错误的 API 协议实现

### 问题分类 GB/T34943-6.2.4.1 堆检查 🟠

**严重级别**: HIGH | **CWE**: [CWE-124](https://cwe.mitre.org/data/definitions/124.html)

使用 realloc() 来调整存储敏感信息的缓冲区。

**漏洞描述**:
使用 realloc() 来调整存储敏感信息的缓冲区会将敏感信息暴露给攻击者，因为这些信息并没有从内存中被删除，攻击者可能使用"堆检查"方法（如内存转储方法）读取堆内存储存的敏感信息。

**修复或规避建议**:
使用 realloc() 函数前先清空该内存块中的敏感信息。

**审计要点**:
- `realloc(ptr, new_size)` 调整含敏感数据的内存前是否先清空
- `memset(ptr, 0, strlen(ptr)); ptr = realloc(ptr, new_size);` 先清零再重分配 ✅
- `memset(ptr, 0, strlen(ptr));` 使用 memset 重置 0

---

### 问题分类 GB/T34943-6.2.4.2 敏感信息存储于上锁不正确的内存空间 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-316](https://cwe.mitre.org/data/definitions/316.html)

将敏感信息存储在被锁定或被错误锁定的内存中。

**漏洞描述**:
程序将敏感信息可能会被虚拟内存管理器从内存写入磁盘的交换文件中，从而使攻击者更容易访问这些敏感信息。

**修复或规避建议**:
选择恰当的平台保护机制锁定存放敏感信息的内存，并检查方法的返回值确保锁定操作执行正确。

**审计要点**:
- `password = (char *) malloc(50 * sizeof(char)); if(password != NULL){ strcpy(password, parameter); }` 密码指向的内存未被锁定
- 是否使用 `VirtualLock(password, 30);` 锁定敏感内存
- 操作完成后是否 `VirtualUnlock(password, 30);` 解锁内存

## 6.2.5 不充分的封装

### 问题分类 GB/T34943-6.2.5.1 公有函数返回私有数组 🟢

**严重级别**: LOW | **CWE**: [CWE-375](https://cwe.mitre.org/data/definitions/375.html)

类定义中的私有数组属性在该类的公有函数中作为返回值返回。

**漏洞描述**:
其中的私有数组作为函数的返回值暴露到了公有区域，存在该私有数组成员在公有区域被修改的风险。

**修复或规避建议**:
当私有数组组成的数据需要作为公有函数的返回值时，应返回该私有数组的副本。

**审计要点**:
- 公有 getter 方法是否直接返回内部私有数组的指针引用
- `return secret;` vs `memcpy(cpy_secret, secret, sizeof(secret)); return cpy_secret;`

## 6.2.7 安全功能

### 问题分类 GB/T34943-6.2.7.1 明文存储口令 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-256](https://cwe.mitre.org/data/definitions/256.html)

明文存储口令，将口令明文存储。

**漏洞描述**:
明文存储口令会降低攻击的难度，攻击者若获得访问口令存储文件的权限，便可轻易获取口令。

**修复或规避建议**:
尽量避免在容易受到攻击的地方存储口令。如果需要，考虑存储口令的加密散列，以替代明文口令存储。

**审计要点**:
- `void store_password(char * password) { void example_fun(char * password){ store_password(password); }` 口令以明文存入数据库
- 是否使用单向哈希存储口令（如 SHA-256+salt）
- 配置文件中是否有明文密码

---

### 问题分类 GB/T34943-6.2.7.2 存储可恢复的口令 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-916](https://cwe.mitre.org/data/definitions/916.html)

采用双向可选的加密算法加密口令并将口令存储在外部文件或数据库中。

**漏洞描述**:
攻击者有机会通过解密算法对口令进行解密还原。

**修复或规避建议**:
当业务不需要取已存储的口令进行还原时，使用单向加密算法对口令进行加密并存储。

**审计要点**:
- AES 对称加密存储口令是可逆的（`aesEncrypto.enc(input, output); store_password(psw);`）
- 应使用 SHA-256 等单向哈希替代可逆加密
- `sha256_enc(input, output); store_password(psw);` ✅ 单向哈希

---

### 问题分类 GB/T34943-6.2.7.3 口令硬编码 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-798](https://cwe.mitre.org/data/definitions/798.html)

程序代码（包括注释）中包含硬编码口令。

**漏洞描述**:
攻击者可以通过反编译或直接读取二进制代码的方式获取硬编码口令。

**修复或规避建议**:
使用单向不可逆的加密算法对口令进行加密并存储在外部文件或数据库中。

**审计要点**:
- `if(strcmp(password, "jfk64k3h6g65w63")) { flag = true; }` 代码中包含硬编码口令
- `SHA256 sha256; sha256.enc(input, output);` 从加密后的字符串比对 ✅

### 问题分类 GB/T34943-6.2.7.4 敏感信息明文传输 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-319](https://cwe.mitre.org/data/definitions/319.html)

敏感信息在传输过程中未进行加密。

**漏洞描述**:
攻击者有机会通过在传输过程中截获或复制报文获取敏感信息。

**修复或规避建议**:
发送敏感信息前对敏感信息进行加密或采用加密通道传输敏感信息。

**审计要点**:
- `send_message(address); //传输 address 是否进行了加密`
- `send_message(address); //传输 address 需进行加密`
- 应使用 HTTPS/TLS 等加密通道传输敏感信息
- 敏感字段（口令、token、身份证号）在传输前是否加密
- 敏感信息传输必须使用 TLS/HTTPS

### 问题分类 GB/T34943-6.2.7.5 使用已破解或危险的加密算法 🟠

**严重级别**: HIGH | **CWE**: [CWE-327](https://cwe.mitre.org/data/definitions/327.html)

软件采用已破解或自定义的非标准加密算法。

**漏洞描述**:
使用已破解或非标准的加密算法是很危险的，因为攻击者有可能破解算法，从而窃取所保护的数据。

**修复或规避建议**:
采用目前加密领域中加密强度较高的标准加密算法。

**审计要点**:
- `DES desEncryptor; unsigned char key[DES_KEYLENGTH];` — DES 已被破解
- `AES_Encryptor aesEncrypto; unsigned char xorBlock[AES_BLOCKSIZE];` — 使用高强度 AES ✅
- 禁用 DES/RC4/Blowfish

### 问题分类 GB/T34943-6.2.7.6 可逆的散列算法 🟠

**严重级别**: HIGH | **CWE**: [CWE-328](https://cwe.mitre.org/data/definitions/328.html)

软件采用的散列算法具有可逆算法。

**漏洞描述**:
该算法可利用生成的散列值确定原始输入，或者可以找到一个输入产生相同的散列值。

**修复或规避建议**:
采用当前公认的不可逆标准散列算法。

**审计要点**:
- `SHA1 sha1; sha1.enc(input, output);` — 使用 SHA1 可逆散列加密算法
- `SHA256 sha256; sha256.enc(input, output);` — 使用 SHA256 标准散列加密算法 ✅
- 密码场景应使用 bcrypt/PBKDF2/Argon2 而非纯 SHA

### 问题分类 GB/T34943-6.2.7.7 密码分组链接模式未使用随机初始化矢量 🟠

**严重级别**: HIGH | **CWE**: [CWE-329](https://cwe.mitre.org/data/definitions/329.html)

密码分组链接模式使用的初始化向量不是一个随机数。

**漏洞描述**:
攻击者有机会通过字典式攻击获取加密的数据。

**修复或规避建议**:
密码分组链接模式使用随机的初始化向量。

**审计要点**:
- `unsigned char key_iv[] = {0x01,0x02,0x03,...}; CBC_Mode(AES)::Encryption Encryptor(key, key_size, iv);` 固定 IV ❌
- 使用 CryptGenRandom 或 CSPRNG 生成随机 IV ✅

---

### 问题分类 GB/T34943-6.2.7.8 不充分的随机数 🟠

**严重级别**: HIGH | **CWE**: [CWE-338](https://cwe.mitre.org/data/definitions/338.html)

软件在安全相关代码中依赖不充分的随机数。

**漏洞描述**:
攻击者可以预测将生成的随机数，绕过依赖随机数的安全保护。

**修复或规避建议**:
使用目前被业界专家认为较强的经过良好审核的加密 PRNG 算法，初始化随机数生成器时使用具有足够长度且不固定的种子。

**审计要点**:
- `unsigned char iv[8]; for(i=0;i<8;i++) { num[i] = rand()%(RAND_MAX+1); }` — 使用不充分的伪随机数生成器
- `HCRYPTPROV hcryptprov; CryptGenRandom(hcryptprov,8,iv);` — 基于加密的伪随机数生成器 CSPRNG ✅

### 问题分类 GB/T34943-6.2.7.9 安全关键的行为依赖反向域名解析 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-350](https://cwe.mitre.org/data/definitions/350.html)

通过反向域名解析获取 IP 地址的域名，然后依赖域名对主机进行身份鉴别。

**漏洞描述**:
攻击者可通过 DNS 欺骗修改 IP 地址与域名的对应关系，从而绕过依赖域名的主机身份鉴别。

**修复或规避建议**:
通过用户名口令、数字证书等其他手段对主机身份进行鉴别。

**审计要点**:
- `gethostbyaddr(my_addr.s_addr, sizeof(in_addr), AF_INET);` 通过域名对主机进行身份鉴别
- 安全决策是否依赖 gethostbyname/gethostbyaddr 反向解析结果

### 问题分类 GB/T34943-6.2.7.10 没有要求使用强口令 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-521](https://cwe.mitre.org/data/definitions/521.html)

软件没有要求用户使用具有足够复杂度的口令。

**漏洞描述**:
攻击者可以很容易地猜出用户的口令或实施暴力破解攻击。

**修复或规避建议**:
要求用户使用具有足够复杂度的口令。口令强度策略应包含下列属性：
- 最小和最大长度；
- 包含字母、数字和特殊字符；
- 不包含用户名；
- 定期更改口令；
- 不使用刚用过的口令；
- 身份鉴别失败一定次数后锁定用户。

**审计要点**:
- `int check_length(string); int check_mode(string); int check_exclude_name(string);` 等口令策略检查
- 注册/修改口令流程是否强制要求强口令策略

### 问题分类 GB/T34943-6.2.7.11 没有对口令域进行掩饰 🟢

**严重级别**: LOW | **CWE**: [CWE-549](https://cwe.mitre.org/data/definitions/549.html)

用户输入口令时没有对口令域进行掩饰。

**漏洞描述**:
增加了攻击者通过观察屏幕获取到口令的可能性。

**修复或规避建议**:
用户输入口令时对 口令域进行掩饰。通常，用户输入的每一个字符都应该以星号形式回显。

**审计要点**:
- `char password[MAX_STR_LEN]; printf("Please enter the password.\n"); while(i < MAX_STR_LEN && (c = getch())) { password[i] = c; i++; ++i; }` 缺少掩码显示
- 应使用 `getch()` 并回显 `*` 号

### 问题分类 GB/T34943-6.2.7.14 RSA算法未使用最优非对称加密填充 🟠

**严重级别**: HIGH | **CWE**: [CWE-780](https://cwe.mitre.org/data/definitions/780.html)

使用 RSA 加密算法时未使用最优非对称加密填充（OAEP）。

**漏洞描述**:
使用 PKCS#1 v1.5 填充的 RSA 加密容易受到 Bleichenbacher 攻击（选择密文攻击），攻击者可通过发送大量精心构造的密文并观察解密结果来逐步恢复明文。未使用 OAEP 填充将显著降低 RSA 加密的安全性。

**修复或规避建议**:
使用 RSA 加密算法时使用最优非对称加密填充（OAEP），如 RSA-OAEP-SHA256。避免使用 PKCS#1 v1.5 填充模式。

**审计要点**:
- 检查 OpenSSL/Crypto++ 等库中 RSA 加密的填充模式设置
- `RSA_public_encrypt(len, from, to, rsa, RSA_PKCS1_PADDING)` — 使用 PKCS1 填充 ❌
- `RSA_public_encrypt(len, from, to, rsa, RSA_PKCS1_OAEP_PADDING)` — 使用 OAEP 填充 ✅
- 检查是否使用了 `RSA_NO_PADDING`（无填充，极度危险）

## 6.2.8 Web 问题

### 问题分类 GB/T34943-6.2.8.1 跨站脚本 🟠

**严重级别**: HIGH | **CWE**: [CWE-79](https://cwe.mitre.org/data/definitions/79.html)

使用未经验证的输入数据构建 Web 页面。

**漏洞描述**:
攻击者可构建任意 Web 页面，并在页面中植入恶意脚本，确保输入数据不影响页面的结构。

**修复或规避建议**:
在构建 Web 页面前对输入数据进行验证或编码，确保输入数据不影响页面的结构。

**审计要点**:
- `char *name = getPar(argv, name); write_html("verification(name)", "index.html");` 输入未经验证直接写入 HTML
- 应对用户输入进行 HTML 实体转义后再输出
- `write_html(verification(name), "index.html");` ✅ 经验证或转义

## 误报排除指南

以下情况通常为误报，不需要修复：

- **工具类/工具函数**：DateUtil、StringUtil、JSONUtil 等通用工具类的方法本身不是漏洞
- **测试代码**：单元测试、集成测试中的代码通常不需要修复
- **示例代码**：Demo、Example 等示例代码
- **配置类**：Properties/yml 配置加载通常不涉及用户输入
- **常量定义**：final static 常量是安全的
- **白名单验证**：已有白名单校验的代码片段
