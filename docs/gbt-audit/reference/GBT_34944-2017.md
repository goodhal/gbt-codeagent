## 6 源代码漏洞测试内容

### 6.1 源代码漏洞分类

本标准根据软件开发中常用的概念来组织 Java 语言源代码漏洞的类别，具体分为如下 **9 个类别**：

| 编号 | 类别 | 说明 |
|------|------|------|
| a) | 行为问题 | 由于应用程序的意外行为而引发的漏洞 |
| b) | 路径错误 | 不恰当的处理访问路径而引发的漏洞 |
| c) | 数据处理 | 在处理数据的功能中发现的问题 |
| d) | 处理程序错误 | 由于未按预期使用 API 而引起的漏洞 |
| e) | 不充分的封装 | 由于缺少封装导致敏感数据泄露 |
| f) | 安全功能 | 软件安全功能如身份鉴别、访问控制、机密性、密码学和特权管理等相关的漏洞 |
| g) | 时间和状态 | 在多系统、进程或线程并发环境下由于对时间和状态管理不当而引发 |
| h) | Web 问题 | Web 技术相关的漏洞 |
| i) | 用户界面错误 | 与用户界面相关的漏洞 |

> 注：Java 语言源代码漏洞的名称及其所属的类别参见附录 B。

## 6.2.1 行为问题

### 问题分类 GB/T34944-6.2.1.1 不可控的内存分配 🟠

**严重级别**: HIGH | **CWE**: [CWE-770](https://cwe.mitre.org/data/definitions/770.html)

内存分配的大小受外部控制的输入数据影响，且程序没有指定内存分配大小的上限。

**漏洞描述**:
攻击者可以使程序分配大量的内存，程序可能会因为内存资源不足而崩溃。

**修复或规避建议**:
在程序中指定内存分配大小的上限，在分配内存前对要分配的内存大小进行验证，确保要分配的内存大小不超过上限。

**审计要点**:
检查是否存在无上限的内存分配（如 `new byte[length]`、`new String[length]`），其中 length 来自用户输入且未做边界检查。

## 6.2.2 路径错误

### 问题分类 GB/T34944-6.2.2.1 不可信的搜索路径 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-426](https://cwe.mitre.org/data/definitions/426.html)

程序使用关键资源时没有指定资源的路径，而是依赖操作系统去搜索资源。

**漏洞描述**:
攻击者可以在搜索优先级更高的文件夹中放入相同名称的资源，程序会使用攻击者控制的资源。

**修复或规避建议**:
使用关键资源时指定资源所在的路径。

**审计要点**:
检查 `Runtime.getRuntime().exec(command)` 中 command 是否包含不受控路径；检查动态加载类/配置文件的路径是否来自用户输入。

## 6.2.3 数据处理

### 问题分类 GB/T34944-6.2.3.1 相对路径遍历 🟠

**严重级别**: HIGH | **CWE**: [CWE-22](https://cwe.mitre.org/data/definitions/22.html)

路径名受外部控制的输入数据影响，且程序没有使能够解析到目录外位置的字符序列（如"../"）失效。

**漏洞描述**:
攻击者可以通过输入能够解析到目录外位置的字符序列来访问限制目录之外的文件或目录。

**修复或规避建议**:
在构建路径名前对输入数据进行验证，确保外部输入仅包含允许的构成路径名的字符。

**审计要点**:
检查文件操作（`new File(basePath + userInput)`）是否过滤了 `../` 和 URL 编码的 `%2e%2e`；`getCanonicalPath()` 后是否检查以基础目录开头。

---

### 问题分类 GB/T34944-6.2.3.2 绝对路径遍历 🟠

**严重级别**: HIGH | **CWE**: [CWE-36](https://cwe.mitre.org/data/definitions/36.html)

路径名由外部控制的输入数据决定，且程序没有限制路径名允许访问的目录。

**漏洞描述**:
攻击者可以通过输入路径名来访问任意的文件或目录。

**修复或规避建议**:
在程序中指定允许访问的文件或目录，在访问文件或目录前对路径名进行验证，确保仅允许访问指定的文件或目录。

**审计要点**:
检查是否存在直接使用用户输入作为绝对路径（如 `new File(absolutePath)`）进行文件操作。

---

### 问题分类 GB/T34944-6.2.3.3 命令注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-78](https://cwe.mitre.org/data/definitions/78.html)

使用未经验证的输入数据构建命令。

**漏洞描述**:
攻击者可执行任意命令。

**修复或规避建议**:
在构建命令前对输入数据进行验证，确保输入数据仅能用于构成允许的命令。

**审计要点**:
检查 `Runtime.getRuntime().exec(cmd)` / `ProcessBuilder` 的参数是否直接拼接用户输入；是否使用了单字符串形式 `exec(String)` 而非安全的 `exec(String[])`；是否有分号/管道符注入风险。

---

### 问题分类 GB/T34944-6.2.3.4 SQL 注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-89](https://cwe.mitre.org/data/definitions/89.html)

使用未经验证的输入数据采用拼接字符串的方式形成 SQL 语句。

**漏洞描述**:
攻击者可植入任何 SQL 语句，实现越权查询数据库的敏感数据、非法修改数据库数据和提升攻击者的数据权限等目的。

**修复或规避建议**:
在拼接 SQL 语句前对输入数据进行验证，确保输入数据不包含 SQL 语句的关键字符；或者采用 PreparedStatement 创建 SQL 语句，并将输入数据作为 SQL 语句的参数。

**审计要点**:
- 是否使用 Statement 拼接 SQL（`"SELECT * FROM user_data WHERE userid = " + owner`）
- 是否使用 PreparedStatement 参数化查询
- MyBatis Mapper XML 或注解中是否使用 `${}` 传递用户输入参数（应使用 `#{}`）
- 排序字段/动态表名是否未做白名单处理

---

### 问题分类 GB/T34944-6.2.3.5 代码注入 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-94](https://cwe.mitre.org/data/definitions/94.html)

使用未经验证的输入数据动态构建代码语句。

**漏洞描述**:
攻击者可构建任意恶意代码，实现远程代码执行（RCE）。

**修复或规避建议**:
在动态构建代码语句前对输入数据进行验证，确保仅能用于构建允许执行的代码。

**审计要点**:
检查 `ScriptEngineManager.getEngineByName("javascript").eval(code)` 等动态代码执行场景；code 参数是否来自用户输入且未做白名单校验。

---

### 问题分类 GB/T34944-6.2.3.6 进程控制 🟠

**严重级别**: HIGH | **CWE**: [CWE-114](https://cwe.mitre.org/data/definitions/114.html)

使用未经验证的输入数据作为动态加载的库的标识符。

**漏洞描述**:
使用未经验证的输入数据作为动态加载库的标识符将使攻击者有机会加载恶意的代码库。

**修复或规避建议**:
在动态加载库前对输入数据进行验证，确保输入数据仅能用于加载允许加载的代码库。

**审计要点**:
检查 `System.load(path)` / `LoadLibrary(libraryName)` 的参数是否来自用户输入；是否有类名白名单限制。

---

### 问题分类 GB/T34944-6.2.3.7 信息通过错误消息泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-209](https://cwe.mitre.org/data/definitions/209.html)

软件呈现给用户的错误消息中包括与环境、用户或相关数据有关的敏感信息。

**漏洞描述**:
对攻击者而言，敏感信息可能本身就是有价值的信息，或者有助于开展进一步的攻击。

**修复或规避建议**:
确保错误消息中仅含有对目标受众有用的少量细节。

**审计要点**:
- 异常堆栈、数据库错误信息、内部路径是否直接返回给客户端
- 错误响应是否包含 SQL 错误信息/表名/字段名
- 是否暴露了服务器版本/框架信息
- catch 块是否只打印 `"Sorry! We will fix the problem soon!"` 等通用消息

---

### 问题分类 GB/T34944-6.2.3.8 信息通过服务器日志文件泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-532](https://cwe.mitre.org/data/definitions/532.html)

将敏感信息写入服务器日志文件。

**漏洞描述**:
攻击者有机会通过访问日志文件读取敏感信息。

**修复或规避建议**:
慎重考虑写入日志文件信息的隐秘性。不要把敏感信息写入日志文件。

**审计要点**:
检查日志记录是否存在密码、Token、身份证号、手机号等敏感字段的输出；`writeLog(msg)` 写入的内容是否包含 username/password。

---

### 问题分类 GB/T34944-6.2.3.9 信息通过调试日志文件泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-532](https://cwe.mitre.org/data/definitions/532.html)

应用程序没有充分限制对用于调式的日志文件的访问。

**漏洞描述**:
调试日志文件通常包含应用程序的敏感信息，攻击者有机会通过访问调试日志文件读取敏感信息。

**修复或规避建议**:
在产品发布之前移除产生日志文件的代码。

**审计要点**:
检查 DEBUG 级别的日志是否在生产环境输出；log4j/logback 配置是否将 DEBUG 日志输出到文件。

---

### 问题分类 GB/T34944-6.2.3.10 信息通过持久 cookie 泄露 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-539](https://cwe.mitre.org/data/definitions/539.html)

将敏感信息存储到持久 cookie 中。

**漏洞描述**:
持久 cookie 是被储存在浏览器所在机器上的硬盘驱动器上的 cookie，攻击者有机会通过访问硬盘上的 cookie 文件获取敏感信息。

**修复或规避建议**:
不要在持久 cookie 中保存敏感信息。

**审计要点**:
检查 cookie 是否设置了 `setMaxAge(14 * 24 * 60 * 60)`（持久化）；Cookie 中是否存储了 username/password 等敏感信息而非 Session ID。

---

### 问题分类 GB/T34944-6.2.3.11 未检查的输入作为循环条件 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-606](https://cwe.mitre.org/data/definitions/606.html)

软件没有对被用于循环条件的输入进行适当的检查。

**漏洞描述**:
攻击者可以让软件循环过多从而使软件拒绝服务。

**修复或规避建议**:
规定循环次数的上限，在将用户输入的数据用于循环条件前验证用户输入的数据是否超过上限。

**审计要点**:
检查循环条件中的变量是否来自用户输入（如 `for(int i = 0; i < count; i++)`）且未做边界检查。

---

### 问题分类 GB/T34944-6.2.3.12 XPath 注入 🟠

**严重级别**: HIGH | **CWE**: [CWE-91](https://cwe.mitre.org/data/definitions/91.html)

使用未经验证的输入动态构造一个 XPath 表达式，来从 XML 数据库中检索数据。

**漏洞描述**:
攻击者可以构造恶意的 XPath 表达式对 XML 数据库进行越权操作。

**修复或规避建议**:
使用参数化的 XPath 查询（例如使用 XQuery）。这将有助于确保数据平面和控制平面相分离。正确地验证用户输入，在适当的情况下拒绝绝数据、过滤数据或对数据进行转义。

**审计要点**:
检查 `XPathExpression expression = com.createExpression(); expression.evaluate(query)` 中 query 是否包含用户输入；是否使用 XPathVariableResolver 实现参数化 XPath。

## 6.2.4 处理程序错误

### 问题分类 GB/T34944-6.2.4.1 未限制危险类型文件的上传 🟠

**严重级别**: HIGH | **CWE**: [CWE-434](https://cwe.mitre.org/data/definitions/434.html)

软件没有限制允许用户上传的文件的类型。

**漏洞描述**:
攻击者可上传具有危险类型的文件，这些文件可能在产品的环境中自动被执行处理。

**修复或规避建议**:
限制允许用户上传的文件的类型。

**审计要点**:
- 是否仅检查文件扩展名（`endsWith(".gif")`）而非 MIME 类型魔数
- 上传目录是否在 webroot 下
- 文件名是否 UUID 重命名
- 是否防止了双扩展名绕过

## 6.2.5 不充分的封装

### 问题分类 GB/T34944-6.2.5.1 可序列化的类包含敏感数据 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-213](https://cwe.mitre.org/data/definitions/213.html)

代码包含一个有敏感数据的类，但是类没有显式的拒绝序列化。

**漏洞描述**:
攻击者可以对该含有敏感数据的类进行序列化，从而使用存储在该里面的敏感数据。

**修复或规避建议**:
定义 `writeObject()` 方法去抛出一个异常，或显式地拒绝序列化。

**审计要点**:
- 实现 Serializable 的类是否有 password/secret/key 等敏感字段未加 transient
- 序列化后是否可能泄露敏感信息
- 是否实现了自定义 `writeObject()` 方法跳过或加密敏感字段

---

### 问题分类 GB/T34944-6.2.5.2 违反信任边界 🟠

**严重级别**: HIGH | **CWE**: [CWE-501](https://cwe.mitre.org/data/definitions/501.html)

数据从不受信的一边移到受信的一边却未经验证。

**漏洞描述**:
程序员更容易错误地相信那些未被验证的数据，导致未经验证的数据被攻击者利用。信任边界可以被认为是程序中的一条线——线的一边是不受信数据，另一边，我们认为是可以被验证的数据穿越安全地带穿过信任边界——从不受信的一边移到受信的一边。

**修复或规避建议**:
增加验证逻辑跨越数据安全地穿过信任边界——从不受信的一边移到受信的一边。

**审计要点**:
- 从请求参数获取的数据存入 session 前是否验证（`session.setAttribute("username", username)` 直接使用用户输入）
- 是否将不受信任的数据释放到 session 中
- 跨信任边界传递的数据是否做了合法性校验

## 6.2.6 安全功能

### 问题分类 GB/T34944-6.2.6.1 明文存储口令 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-256](https://cwe.mitre.org/data/definitions/256.html)

将口令明文存储。

**漏洞描述**:
明文存储口令会降低攻击的难度，攻击者若获得访问口令存储文件的权限，便可轻易获取口令。

**修复或规避建议**:
尽量避免在容易受到攻击的地方存储口令。如果需要，考虑存储口令的加密散列，以替代明文口令存储。

**审计要点**:
- 数据库 password 字段是否存储明文或 Base64 编码
- 是否使用了可逆加密（AES 加密密码）而非单向哈希
- properties 配置文件中是否有明文密码

---

### 问题分类 GB/T34944-6.2.6.2 存储可恢复的口令 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-916](https://cwe.mitre.org/data/definitions/916.html)

采用双向可选的加密算法加密口令并将口令存储在外部文件或数据库中。

**漏洞描述**:
攻击者有机会通过解密算法对口令进行解密还原。

**修复或规避建议**:
当业务不需要取已存储的口令进行还原时，使用单向加密算法对口令进行加密并存储。

**审计要点**:
检查口令存储是否使用双向可逆加密（应使用 bcrypt/scrypt/PBKDF2 等单向哈希）。

---

### 问题分类 GB/T34944-6.2.6.3 口令硬编码 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-798](https://cwe.mitre.org/data/definitions/798.html)

程序代码（包括注释）中包含硬编码口令。

**漏洞描述**:
攻击者可以通过反编译或直接读取二进制代码的方式获取硬编码口令。

**修复或规避建议**:
使用单向不可逆的加密算法对口令进行加密并存储在外部文件或数据库中。

**审计要点**:
检查源代码中是否存在硬编码的密码、密钥、API Token（如 `if ("0123dfgdd4835".equals(password))`）。

---

### 问题分类 GB/T34944-6.2.6.4 依赖 referer 字段进行身份鉴别 🟠

**严重级别**: HIGH | **CWE**: [CWE-293](https://cwe.mitre.org/data/definitions/293.html)

依赖 HTTP 请求的 referer 字段进行身份鉴别。

**漏洞描述**:
攻击者可以通过修改 HTTP 请求的 referer 字段设置冒用其他用户的身份。Referer 头是由客户端浏览器发送的，攻击者可以轻易伪造或删除该字段，导致身份鉴别机制被完全绕过。

**修复或规避建议**:
通过用户名口令、数字证书、Session Token 等其他可靠手段对用户身份进行鉴别，不应依赖可被客户端控制的 HTTP 头信息。

**审计要点**:
检查身份鉴别逻辑是否依赖 `request.getHeader("referer")` 进行判断；referer 是否容易被伪造；是否存在基于 Referer 的访问控制逻辑。

---

### 问题分类 GB/T34944-6.2.6.5 Cookie 中的敏感信息明文存储 🟠

**严重级别**: HIGH | **CWE**: [CWE-315](https://cwe.mitre.org/data/definitions/315.html)

敏感信息在 cookie 中明文存储。

**漏洞描述**:
攻击者可通过 XSS 攻击、中间人攻击或物理访问等方式获取 Cookie 中的敏感信息，导致用户身份被盗用或敏感数据泄露。

**修复或规避建议**:
将需要存储到 cookie 中的敏感信息先加密后再存储。

**审计要点**:
检查 Cookie 值是否直接存储了 address、username、password 等明文敏感信息。

---

### 问题分类 GB/T34944-6.2.6.6 敏感信息明文传输 🔴

**严重级别**: CRITICAL | **CWE**: [CWE-319](https://cwe.mitre.org/data/definitions/319.html)

敏感信息在传输过程中未进行加密。

**漏洞描述**:
攻击者有机会通过在传输过程中截获或复制报文获取敏感信息。

**修复或规避建议**:
发送敏感信息前对敏感信息进行加密或采用加密通道传输敏感信息。

**审计要点**:
检查敏感信息（account、password 等）的传输是否使用 TLS/HTTPS 加密；HTTP 接口中是否明文传输密码/token。

---

### 问题分类 GB/T34944-6.2.6.7 使用已破解或危险的加密算法 🟠

**严重级别**: HIGH | **CWE**: [CWE-327](https://cwe.mitre.org/data/definitions/327.html)

软件采用已破解或自定义的非标准加密算法。

**漏洞描述**:
使用已破解或非标准的加密算法是很危险的，因为攻击者有可能破解算法，从而窃取所保护的数据。

**修复或规避建议**:
采用目前加密领域中加密强度较高的标准加密算法。

**审计要点**:
- `Cipher.getInstance("DES")` — 使用加密强度较低的 DES 算法
- 应使用 AES/GCM/NoPadding 或 AES/CBC/PKCS5Padding
- 禁用 DES/3DES/RC4/RC2/Blowfish

---

### 问题分类 GB/T34944-6.2.6.8 可逆的散列算法 🟠

**严重级别**: HIGH | **CWE**: [CWE-328](https://cwe.mitre.org/data/definitions/328.html)

软件采用的散列算法具有可逆算法。

**漏洞描述**:
该算法可利用生成的散列值确定原始输入，或者可以找到一个输入产生相同的散列值。

**修复或规避建议**:
采用当前公认的不可逆标准散列算法。

**审计要点**:
- `MessageDigest.getInstance("SHA-1")` — 使用 SHA-1 散列加密算法
- `MessageDigest.getInstance("SHA-256")` — 使用 SHA-256 标准散列加密算法
- 密码哈希应使用 BCrypt/PBKDF2/Argon2 而非纯 SHA 系列

---

### 问题分类 GB/T34944-6.2.6.9 密码分组链接模式未使用随机初始化矢量 🟠

**严重级别**: HIGH | **CWE**: [CWE-329](https://cwe.mitre.org/data/definitions/329.html)

密码分组链接模式使用的初始化向量不是一个随机数。

**漏洞描述**:
攻击者有机会通过字典式攻击获取加密的数据。

**修复或规避建议**:
密码分组链接模式使用随机的初始化向量。

**审计要点**:
检查 CBC 模式加密是否使用固定 IV（如 `byte[] iv = {0x00,0x00,0x00,0x00,...,0x00}`）；IV 必须每次加密使用 SecureRandom 随机生成。

---

### 问题分类 GB/T34944-6.2.6.10 不充分的随机数 🟠

**严重级别**: HIGH | **CWE**: [CWE-338](https://cwe.mitre.org/data/definitions/338.html)

软件在安全相关代码中依赖不充分的随机数。

**漏洞描述**:
攻击者可以预测将生成的随机数，绕过依赖随机数的安全保护。

**修复或规避建议**:
使用目前被业界专家认为较强的经过良好审核的加密 PRNG 算法，初始化随机数生成器时使用具有足够长度且不固定的种子。

**审计要点**:
- `byte[] iv = new byte[8]; Random random = new Random()` — 不充分的伪随机数生成器
- 应使用 `SecureRandom` 替代 `java.util.Random`
- `Random random = new SecureRandom()` ✅

---

### 问题分类 GB/T34944-6.2.6.11 安全关键的行为依赖反向域名解析 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-350](https://cwe.mitre.org/data/definitions/350.html)

通过反向域名解析获取 IP 地址的域名，然后依赖域名对主机进行身份鉴别。

**漏洞描述**:
攻击者可通过 DNS 欺骗修改 IP 地址与域名的对应关系，从而绕过依赖域名的主机身份鉴别。

**修复或规避建议**:
通过用户名口令、数字证书等其他手段对主机身份进行鉴别。

**审计要点**:
检查安全决策是否依赖 `InetAddress.getCanonicalHostName().endsWith("trustme.com")` 的反向解析结果。

---

### 问题分类 GB/T34944-6.2.6.12 关键参数篡改 🟠

**严重级别**: HIGH | **CWE**: [CWE-807](https://cwe.mitre.org/data/definitions/807.html)

软件未经经验证使用可能被篡改的关键参数（如资产数据等）。

**漏洞描述**:
某些数据如页面隐藏域、Cookies、URL 等所集传递入的关键参数可能被恶意的混淆器件、代理服务器等方式篡改。若程序获取该类数据后不经验证直接使用，可能造成信息泄露、数据异常、系统路由异常等事故。

**修复或规避建议**:
将关键参数保留到服务端的会话中，程序使用该数据须通过会话获取。

**审计要点**:
- 价格/数量等关键参数是否从 request.getParameter 直接获取而不经服务端校验
- 是否使用 encode 函数做简单的混淆但实际可被绕过
- 关键业务参数是否应在服务端维护并通过 session 获取

---

### 问题分类 GB/T34944-6.2.6.13 没有要求使用强口令 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-521](https://cwe.mitre.org/data/definitions/521.html)

软件没有要求用户使用具有足够复杂度的口令。

**漏洞描述**:
攻击者可以很容易地猜出用户的口令或实施暴力破解攻击。

**修复或规避建议**:
要求用户使用具有足够复杂度的口令。口令强度策略应包含下列属性：
- 最小和最大长度；
- 包含字母、数字和特殊字符；
- 不包含用户名；
- 定期更改口令；
- 不使用刚用过的口令；
- 身份鉴别失败一定次数后锁定用户。

**审计要点**:
检查注册/修改口令的流程是否强制要求强口令策略；是否检查口令长度、复杂度、是否包含用户名等。

---

### 问题分类 GB/T34944-6.2.6.14 没有对口令域进行掩饰 🟢

**严重级别**: LOW | **CWE**: [CWE-549](https://cwe.mitre.org/data/definitions/549.html)

用户输入口令时没有对口令域进行掩饰。

**漏洞描述**:
增加了攻击者通过观察屏幕获取到口令的可能性。

**修复或规避建议**:
用户输入口令时对 口令域进行掩饰。通常，用户输入的每一个字符都应该以星号形式回显。

**审计要点**:
检查口令输入框是否设置掩码或密码模式（如 JPasswordField.setEchoChar('*')）。

---

### 问题分类 GB/T34944-6.2.6.15 依赖未经验证和完整性检查的 cookie 🟠

**严重级别**: HIGH | **CWE**: [CWE-565](https://cwe.mitre.org/data/definitions/565.html)

应用程序在执行与安全非常关键的操作时，依赖 Cookies 的存在或它的值，而又没有正确确保这些设置对相关的用户是有效的。

**漏洞描述**:
攻击者可以很容易的在浏览器中或浏览器外的客户端代码修改 cookie。依赖未经验证和完整性检查的 cookie 可以允许攻击者绕过验证、执行像 SQL 注入这样的注入攻击和跨站脚本攻击或者以修改的方式篡改输入。

**修复或规避建议**:
做一个安全无关的决定时，依赖服务器端存储的数据。避免依赖客户端传过来的 Cookie 数据。

**审计要点**:
- Cookie 中的 isAdmin/isAdmin 等角色标识是否可直接伪造
- 是否仅凭 Cookie 判断管理员身份而无服务端验证
- Cookie 值是否经过签名或完整性保护

---

### 问题分类 GB/T34944-6.2.6.16 通过用户控制的 SQL 关键字绕过授权 🟠

**严重级别**: HIGH | **CWE**: [CWE-89](https://cwe.mitre.org/data/definitions/89.html)

软件使用的数据库表其中包含了某个用户不应有权访问的记录，但该软件执行的一个 SQL 语句中的关键字段可以受用户控制。

**漏洞描述**:
如果用户可以将关键字设置为任何值，那么该用户就可以修改该关键字指向未经授权的记录，实现水平越权访问其他用户的数据。

**修复或规避建议**:
对用户输入的关键字进行验证，确保其只能访问该用户有权访问的记录。使用白名单验证动态列名和表名。

**审计要点**:
- ORDER BY 排序字段是否由用户输入控制
- 动态表名/列名是否未做白名单处理
- MyBatis `${}` 用于表名/列名时的注入风险
- 是否存在通过修改查询条件访问其他用户数据的可能

---

### 问题分类 GB/T34944-6.2.6.17 HTTPS 会话中的敏感 cookie 没有设置安全性 🟠

**严重级别**: HIGH | **CWE**: [CWE-614](https://cwe.mitre.org/data/definitions/614.html)

HTTPS 会话中的敏感 cookie 没有设置安全属性。

**漏洞描述**:
若没有设置安全属性，敏感 cookie 可通过 HTTP 会话以明文形式发送，可能被攻击者截获。

**修复或规避建议**:
设置 HTTPS 会话中的敏感 cookie 的安全属性。

**审计要点**:
- `response.addCookie(cookieUserID)` 后是否调用 `cookieID.setSecure(true)`
- HTTPS 环境下 Cookie 是否设置了 secure 属性
- 是否设置了 HttpOnly 属性防止 XSS 窃取

---

### 问题分类 GB/T34944-6.2.6.18 未使用盐值计算散列值 🟠

**严重级别**: HIGH | **CWE**: [CWE-759](https://cwe.mitre.org/data/definitions/759.html)

软件针对口令等不应可选的输入使用了单向加密数列，但未使用盐值。

**漏洞描述**:
如果未使用盐值计算散列值，攻击者就更容易利用彩虹表等字典式攻击技术破解口令。

**修复或规避建议**:
使用盐值计算散列值，增加攻击者破解口令的难度。

**审计要点**:
- 密码哈希是否使用随机盐值（salt）
- PBKDF2KeySpec / sendSalt 流程是否正确生成和使用盐值
- 盐值是否与哈希结果一同存储

## 6.2.7 时间和状态

### 问题分类 GB/T34944-6.2.7.1 会话固定 🟠

**严重级别**: HIGH | **CWE**: [CWE-384](https://cwe.mitre.org/data/definitions/384.html)

应用程序在进行身份鉴别并建立一个新的会话时没有让原来的会话失效。

**漏洞描述**:
攻击者可使用户在攻击者创建的会话基础上进行身份鉴别，从而窃取用户通过身份鉴别的会话并冒充用户进行恶意操作。

**修复或规避建议**:
对用户进行身份鉴别并建立一个新的会话时让原来的会话失败。

**审计要点**:
- 登录成功后是否调用 `session.invalidate()` 并重建新 session
- 是否存在 Session ID 可被 URL 参数控制的情况
- 登录前后 session ID 是否变化

---

### 问题分类 GB/T34944-6.2.7.2 会话永不超时 🟡

**严重级别**: MEDIUM | **CWE**: [CWE-613](https://cwe.mitre.org/data/definitions/613.html)

设置会话为永不超时。

**漏洞描述**:
攻击者可以有时间重复使用原有的会话凭据或会话 TID 来进行授权。

**修复或规避建议**:
设置会话超时的日期。

**审计要点**:
- `<session-timeout>-1</session-timeout>` 表示值为 -1 即 session 永不过期
- 是否同时实施了绝对超时（如 8 小时）
- 敏感操作前是否重新验证身份

## 误报排除指南

以下情况通常为误报，不需要修复：

- **工具类/工具函数**：DateUtil、StringUtil、JSONUtil 等通用工具类的方法本身不是漏洞
- **测试代码**：单元测试、集成测试中的代码通常不需要修复
- **示例代码**：Demo、Example 等示例代码
- **配置类**：Properties/yml 配置加载通常不涉及用户输入
- **常量定义**：final static 常量是安全的
- **白名单验证**：已有白名单校验的代码片段
