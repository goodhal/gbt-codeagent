## 🔴 全部代码文件审计强制要求

**⚠️ 极其重要**：LLM审计必须覆盖项目中的**全部源代码文件**，不得遗漏任何文件。

**禁止行为**：
- ❌ 仅审计"关键代码文件"（不同LLM对"关键"的理解不一致）
- ❌ 仅审计Controller/Service层（漏洞可能存在于任何文件）
- ❌ 跳过配置文件、工具类、实体类等

**正确做法**：
- ✅ 使用Glob工具获取全部代码文件列表
- ✅ 逐个阅读并审计每个文件
- ✅ 文件覆盖率达到100%

**执行示例**：
```bash
# 获取全部Java文件
Glob path="项目目录" pattern="**/*.java"

# 逐个审计（假设返回122个文件）
Read file_path="文件1.java" limit=200
Read file_path="文件2.java" limit=200
...（必须审计全部122个文件）
```

## 🔧 三层审计分工

根据审计结果，正确的分工应该是：

| 分工 | 负责方 | 发现的漏洞类型 | 检出率 |
|------|--------|---------------|--------|
| **快速扫描** | 代码（正则表达式） | 高风险函数调用（命令注入、SQL注入、缓冲区溢出等） | ~64% |
| **LLM审计** | LLM（语义分析） | 需要上下文分析的漏洞（业务逻辑、输入验证、认证安全等） | ~61% |
| **LLM审查** | LLM（深入分析） | 复杂业务逻辑漏洞、漏洞验证、最终决策 | 最高 |

### 第一层：快速扫描（代码负责）

**职责**：使用正则表达式搜索高风险函数，发现高风险函数调用

**⚠️ 重要说明**：快速扫描由代码自动执行，LLM审计**不应重复**快速扫描的工作。

**快速扫描覆盖的漏洞类型**：
- 命令注入：`os.system`, `Runtime.exec`, `system()` 等
- SQL注入：SQL拼接模式
- 缓冲区溢出：`strcpy`, `gets`, `sprintf` 等
- 硬编码凭证：`password = "..."`, `HARDCODED_*` 等
- 弱加密：`MD5`, `SHA1`, `DES` 等
- 反序列化：`pickle.load`, `yaml.load` 等
- 路径遍历：`new File(path)`, `File.ReadAllText()` 等
- 信息泄露：`println.*password`, `logging.*password` 等

### 第二层：LLM审计（LLM负责）

**职责**：语义分析，发现需要上下文分析的漏洞

**⚠️ 重要说明**：LLM审计应专注于需要上下文分析的漏洞，不应使用关键字搜索。

**LLM审计覆盖的漏洞类型**：
- 输入验证问题：关键状态数据被外部可控、数据真实性验证不足等
- 业务逻辑问题：条件比较不充分、条件语句缺失默认情况、死代码等
- 认证安全问题：身份鉴别过程暴露多余信息、身份鉴别被绕过等
- 并发安全问题：未加限制的外部可访问锁、共享资源的并发安全等
- 会话安全问题：不同会话间信息泄露、发布未完成初始化的对象等
- 内存安全问题：忽略字符串串尾符、对环境变量长度做出假设等
- 资源安全问题：重复释放资源、资源或变量不安全初始化等

**🔴 LLM审计强制要求**：
- 必须审计全部代码文件，不得遗漏任何文件
- 必须检查每个Controller的每个接口方法
- 必须检查每个Service的业务逻辑
- 必须检查每个配置类（Security、Cors、Session等）
- 必须检查每个工具类的输入验证逻辑
- 必须检查每个实体类的敏感字段处理

### 🔴 LLM审计详细检查清单

**每个文件必须检查以下内容**：

#### 1. Controller层检查
- [ ] 每个接口方法是否验证用户输入？
- [ ] 是否有权限检查（@PreAuthorize、SecurityContextHolder）？
- [ ] 是否有CSRF Token验证？
- [ ] 是否有速率限制？
- [ ] 重定向目标是否白名单验证？
- [ ] 文件操作是否路径遍历防护？
- [ ] SQL查询是否参数化？
- [ ] 日志记录是否脱敏？

#### 2. Service层检查
- [ ] 业务逻辑是否有状态绕过风险？
- [ ] 并发操作是否有竞态条件？
- [ ] 金额计算是否使用BigDecimal？
- [ ] 是否有整数溢出风险？
- [ ] 是否有事务管理？

#### 3. 配置类检查
- [ ] Security配置是否启用CSRF保护？
- [ ] CORS配置是否限制可信源？
- [ ] Session配置是否安全？
- [ ] 密码编码器是否使用单向哈希？
- [ ] 是否启用HSTS？

#### 4. 工具类检查
- [ ] 输入验证是否充分？
- [ ] 随机数生成是否使用SecureRandom？
- [ ] 加密算法是否安全？
- [ ] 文件操作是否路径验证？

#### 5. 实体类检查
- [ ] 敏感字段是否标记transient？
- [ ] 序列化是否安全？
- [ ] 是否有敏感数据暴露？

## 🎯 LLM 审计分层策略

**⚠️ 强制要求**：LLM 审计必须按照以下分层策略执行，确保全面覆盖。

### 第一轮：高风险函数扫描（由快速扫描负责）

**⚠️ 重要说明**：此轮审计由快速扫描通过模式匹配自动执行。LLM审计应通过漏洞类型详解文件进行上下文分析。

**目标**：快速定位高风险代码区域

**快速扫描覆盖的漏洞类型**：
| 漏洞类型 | 快速扫描检测方式 |
|---------|----------------|
| 命令注入 | `os.system`, `Runtime.exec`, `system()` 等 |
| SQL注入 | SQL拼接模式 |
| 缓冲区溢出 | `strcpy`, `gets`, `sprintf` 等 |
| 硬编码凭证 | `password = "..."`, `HARDCODED_*` 等 |
| 弱加密 | `MD5`, `SHA1`, `DES` 等 |
| 反序列化 | `pickle.load`, `yaml.load` 等 |
| 路径遍历 | `new File(path)`, `File.ReadAllText()` 等 |
| 信息泄露 | `println.*password`, `logging.*password` 等 |

**LLM审计应**：结合漏洞类型详解文件进行上下文分析，验证快速扫描结果，分析业务逻辑上下文。

---

### 第二轮：数据流分析（LLM审计负责）

**目标**：追踪用户输入到危险函数的路径

```
用户输入追踪：
1. 识别用户输入点：HTTP参数、环境变量、配置文件
2. 追踪数据传递：变量赋值、函数调用、类属性传递
3. 定位危险使用：命令执行、SQL拼接、文件操作、输出操作

示例流程：
HTTP Request → 参数提取 → 变量赋值 → 函数传递 → 危险函数调用
```

**检查要点**：
- 用户输入是否经过验证/过滤？
- 敏感数据是否经过加密/脱敏？
- 配置数据是否影响安全决策？

### 第三轮：业务逻辑审计

**目标**：发现业务逻辑缺陷和认证授权问题

| 审计领域 | 检查要点 | 常见问题 |
|---------|---------|---------|
| **认证流程** | 登录、注销、密码重置、会话管理 | 认证绕过、会话固定、密码重置漏洞 |
| **授权检查** | 权限验证、访问控制、角色管理 | 水平越权、垂直越权、权限缺失 |
| **会话管理** | session创建、超时、固定、并发 | 会话固定、会话劫持、并发登录 |
| **状态转换** | 订单状态、支付流程、审批流程 | 状态绕过、流程篡改、条件竞争 |

**检查技巧**：
- 分析认证流程的每个分支，寻找绕过路径
- 检查权限验证是否在所有敏感操作前执行
- 验证会话管理是否符合安全最佳实践
- 分析状态转换是否有不合理的跳转

### 第四轮：组合漏洞分析

**目标**：发现多个弱点组合形成的攻击链

| 组合模式 | 攻击链 | 最终影响 |
|---------|--------|---------|
| **信息泄露 + 身份伪造** | 泄露用户信息 → 伪造身份 → 账户接管 | 账户被盗 |
| **SSRF + 内网访问** | SSRF漏洞 → 访问内网 → 权限提升 | 内网渗透 |
| **路径遍历 + 文件上传** | 路径遍历 → 上传恶意文件 → RCE | 服务器被控 |
| **XSS + CSRF** | 窃取Cookie → 伪造请求 → 执行操作 | 账户被盗 |
| **注入 + 权限绕过** | SQL注入 → 获取权限 → 越权操作 | 数据泄露 |

**分析技巧**：
- 识别单个漏洞的利用前提
- 分析漏洞之间的依赖关系
- 评估组合攻击的可行性和影响

## 🔧 语言特定审计指导

## 行号验证流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LLM 审计行号验证流程                              │
├─────────────────────────────────────────────────────────────────────┤
│  1️⃣ 读取源文件  ◀ 【必须】使用 Read 工具读取完整源文件              │
│  2️⃣ 定位问题代码  ◀ 【必须】在源文件中精确搜索问题代码片段          │
│  3️⃣ 使用 Grep 验证  ◀ 【必须】使用 Grep 工具精确验证行号           │
│  4️⃣ 确认行号  ◀ 【必须】验证行号对应的是代码行而非注释行            │
│  5️⃣ 记录发现  ◀ 【必须】行号验证通过后才创建发现对象               │
└─────────────────────────────────────────────────────────────────────┘
```

### 🔴 LLM 强制要求

**必须执行**：
- 🔴 **必须使用 Grep 工具验证行号**（禁止凭记忆填写）
- 🔴 **必须逐行核对源文件确认漏洞位置**
- 🔴 **行号必须精确对应问题代码所在行**
- 🔴 **禁止根据注释推断行号，必须定位实际代码**

**验证步骤**：
```bash
# 1. 读取源文件
Read file_path="目标文件" offset=预估行号-10 limit=20

# 2. 使用 Grep 精确查找
Grep pattern="问题代码关键字" path="目标文件" output_mode="content" -n

# 3. 确认行号对应实际代码
# 检查该行是否为代码行（非注释行、非空行）
```

### ⚠️ ValidationService 自动验证

**ValidationService 会做什么**：
- ✅ 自动去重（按 `file:line:type` 去重）
- ✅ 自动验证代码片段是否存在
- ✅ 自动修正错误的行号（±10 行搜索）
- ✅ 使用 Grep 工具精确验证行号
- ✅ 验证通过后状态设为"有效"
- ✅ 验证失败记录为幻觉

**工作流程**：
```
LLM创建发现对象 → ValidationService 自动验证 →
验证通过：状态设为"有效"，出现在报告中
验证失败：记录为幻觉，不出现在报告中
```

### ❌ 常见错误

| 错误类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| **不验证行号** | 凭记忆填写行号 | 必须使用 Grep 工具验证 |
| **注释行号** | 报告行号27，实际是注释行 | 必须定位到实际代码行 |
| **行号偏移** | 报告行号150，实际代码在133 | 使用 Grep 精确搜索 |

## ValidationService 验证流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    ValidationService 验证流程                        │
├─────────────────────────────────────────────────────────────────────┤
│  1️⃣ 加载所有发现对象  ◀ 快速扫描 + LLM 审计                         │
│  2️⃣ 去重处理  ◀ 按 file:line:type 去重，LLM 审计优先                │
│  3️⃣ 验证状态为"误报"的发现  ◀ 代码片段验证 + 行号修正               │
│  4️⃣ 状态更新  ◀ 验证通过 → "有效"；验证失败 → 幻觉                  │
│  5️⃣ 生成报告  ◀ 只包含状态为"有效"的发现                           │
└─────────────────────────────────────────────────────────────────────┘
```

**⚠️ 去重和验证由代码自动并行执行**：

- **去重**：内存操作，按 `file:line:type` 去重，LLM 审计优先
- **验证**：使用 Promise.all 并行验证代码片段，自动修正行号
- **Grep 验证**：使用 Grep 工具精确验证行号

```
┌─────────┬─────────┬─────────┬─────────┐
│ 线程 1  │ 线程 2  │ 线程 3  │ 线程 4  │  ← Promise.all 并行验证
│ 批次 1  │ 批次 2  │ 批次 3  │ 批次 4  │
│ 文件1-20│文件21-40│文件41-60│文件61-80│
└─────────┴─────────┴─────────┴─────────┘
    ↓
合并结果 → 生成报告
```

> **注意**：此步骤由 `ValidationService` 自动执行，无需 LLM 介入。

## 常见幻觉问题

| 幻觉类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| **行号幻觉** | 凭记忆填写行号，实际代码在别处 | 使用 Grep 精确搜索 |
| **代码片段幻觉** | 报告代码片段不存在于文件中 | 使用 Read 工具读取确认 |
| **漏洞类型幻觉** | 将安全代码误判为漏洞 | 分析代码上下文，确认是否有防护措施 |
| **严重程度幻觉** | 将低危漏洞报告为高危 | 根据实际危害评估严重程度 |

## 审计覆盖率检查

**⚠️ 重要说明**：审计完成后，应使用审计覆盖率检查工具验证审计完整性。

**执行方式**：
```bash
# 使用审计覆盖率检查工具
node src/tools/auditCoverageChecker.js --target <项目目录> --findings <发现目录>
```

**检查内容**：
- 文件覆盖率：审计的文件数 / 总文件数
- 语言覆盖率：审计的语言数 / 检测到的语言数
- 漏洞类型覆盖率：审计的漏洞类型数 / 应审计的漏洞类型数

> 详细检查方法见 `src/tools/auditCoverageChecker.js`
