# 输出质量检查标准

本文档定义审计报告的质量检查规则，防止敷衍内容。

## 修复方案编写要求

### ❌ 禁止以下敷衍内容（出现则验证失败）

1. "根据国标要求修复"
2. "消除安全隐患"
3. "使用安全的方法"
4. "加强验证"
5. "进行过滤"
6. "根据相关标准处理"
7. "修复安全漏洞"
8. 字数 < 30 字
9. 不包含具体代码、命令、配置或 API 名称
10. 不包含可执行的技术方案

### ✅ 必须包含以下内容

1. **具体代码示例**（如 `PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?");`）
2. **或具体命令**（如 `chmod 600 /etc/secret`）
3. **或具体 API/类名**（如 `Cipher.getInstance("AES/GCM/NoPadding")`、`BCrypt.hashpw()`）
4. **或具体配置参数**（如 `-fstack-protector`、`TLS 1.2+`）
5. **字数 ≥ 30 字**
6. **必须包含技术动作词**：如"使用"、"改用"、"替代"、"配置"、"启用"等

## 修复方案示例库

| 漏洞类型 | ✅ 合格修复方案 | ❌ 不合格修复方案 |
|---------|----------------|-----------------|
| SQL 注入 | 使用 PreparedStatement：`ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?"); ps.setString(1, userId);` 禁止字符串拼接 SQL。 | 根据国标要求修复，消除安全隐患。 |
| 命令注入 | 改用 ProcessBuilder 参数数组：`new ProcessBuilder("cat", filename).start()` 并对 filename 白名单验证仅允许字母数字。 | 使用安全的方法执行命令。 |
| 硬编码密码 | 密码移至环境变量：`System.getenv("DB_PASSWORD")` 或使用配置文件（权限 600），口令存储使用 bcrypt 哈希。 | 加强密码管理。 |
| 弱加密 | 使用 AES-256/GCM 替代 DES：`Cipher.getInstance("AES/GCM/NoPadding")` 密钥至少 256 位，使用 SecureRandom 生成。 | 使用更安全的加密算法。 |
| 弱哈希 | 使用 SHA-256 或 SM3 替代 MD5/SHA-1，口令哈希使用 bcrypt：`BCrypt.hashpw(password, BCrypt.gensalt(12))`。 | 进行过滤。 |
| 缓冲区溢出 | 使用 strncpy 替代 strcpy：`strncpy(buf, input, sizeof(buf)-1); buf[sizeof(buf)-1] = '\0';` 启用编译器保护 `-fstack-protector`。 | 加强边界检查。 |
| 反序列化 | 禁止 pickle.loads() 反序列化不受信任数据，改用 JSON，或实现白名单类限制可反序列化的类。 | 进行验证。 |

## 质量检查清单

> **🔴 使用时机：创建每个 md 文件时必须执行此检查**
> - 在写入 md 文件之前，逐项核对以下清单
> - 不符合要求的发现不应创建 md 文件
> - 这是 LLM 的责任，代码无法判断语义质量

**每个发现必须满足**：

- [ ] **问题描述**：字数≥20，说明了漏洞原因、攻击方式、潜在风险
- [ ] **修复方案**：字数≥30，提供了具体修复方法、修复原理、代码示例或API建议
- [ ] **严重等级**：符合漏洞实际危害（考虑上下文、可利用性、影响范围）
- [ ] **国标映射**：格式正确，双映射（Java/C++/C#）或单映射（其他语言）
- [ ] **行号验证**：使用 Grep 工具验证，确保准确

## 正确使用工具顺序

| 步骤 | 工具 | 目的 |
|------|------|------|
| 1. 阅读源代码 | Read | 理解代码上下文、业务逻辑 |
| 2. 发现漏洞 | - | 通过语义分析发现需要上下文的漏洞 |
| 3. 有必要时验证行号 | Grep | 确认漏洞位置准确（基于理解后使用） |
| 4. 创建md文件 | Write | 记录审计发现 |

> **说明**：Grep工具可以使用，但应基于对代码的理解后，有必要时才使用（如验证行号、确认细节），不应直接用关键字搜索来发现漏洞。

## 双保险验证机制

### 第一道：LLM 自主检查（创建 md 时执行）

> **🔴 使用时机：创建每个 md 文件之前**
> - 在调用 Write 工具写入 md 文件之前，必须逐项核对上述"质量检查清单"
> - 不符合要求的发现不应创建 md 文件，应重新编写内容
> - 这是 LLM 的责任，代码无法判断语义质量（如"问题描述是否说明了漏洞原因"）

### 第二道：finalize_report 自动验证（生成报告时执行）

> **时机说明**：步骤 6 调用 finalize_report 时自动执行，无需 LLM 介入

`finalize_report` 会自动验证所有发现：
- 去重处理：按 `file:line:type` 去重
- 代码片段验证：验证代码片段是否存在
- 行号修正：自动修正错误的行号
- 状态更新：验证通过 → "有效"；验证失败 → 幻觉

```
【步骤 5 完成确认】

- 已创建：X 个 llm_audit md 文件
- 状态判定：已判定 baseline md 文件状态（有效/误报）
- 独立性：未参考快速扫描结果，逐行阅读源代码
- 下一步：步骤 6 - 生成报告
```

### 步骤 6（生成报告）完成确认

```
【步骤 6 完成确认】
- 已执行：finalize_report 自动验证

- 报告生成：只包含状态为"有效"的发现
- 下一步：步骤 7 - 审计完成
```
