## 🔴 质量检查清单（创建发现时必须核对）

**每个发现对象必须满足以下要求**：

### 1. 必填字段完整性

| 字段 | 是否必填 | 说明 | 示例 |
|------|---------|------|------|
| `source` | ✅ 必填 | 发现来源 | `"llm"` |
| `skillId` | ✅ 必填 | 技能 ID | `"gbt-code-audit"` |
| `vulnId` | ✅ 必填 | 漏洞类型 | `"AUTH_BYPASS"` |
| `title` | ✅ 必填 | 漏洞标题 | `"发现认证绕过漏洞"` |
| `severity` | ✅ 必填 | 严重等级 | `"high"` / `"medium"` / `"low"` |
| `severityLabel` | ✅ 必填 | 严重等级标签 | `"高危"` / `"中危"` / `"低危"` |
| `confidence` | ✅ 必填 | 置信度 | `0.85`（0.0-1.0） |
| `location` | ✅ 必填 | 文件路径和行号 | `"src/controllers/AuthController.java:45"` |
| `file` | ✅ 必填 | 文件路径 | `"src/controllers/AuthController.java"` |
| `line` | ✅ 必填 | 行号 | `45` |
| `vulnType` | ✅ 必填 | 漏洞类型 | `"AUTH_BYPASS"` |
| `cwe` | ✅ 必填 | CWE 编号 | `"CWE-287"` |
| `gbtMapping` | ✅ 必填 | 国标映射 | `"GB/T34944-6.3.1.2 身份鉴别被绕过；GB/T39412-6.3.1.2 身份鉴别被绕过"` |
| `cvssScore` | ✅ 必填 | CVSS 评分 | `8.0`（0.0-10.0） |
| `language` | ✅ 必填 | 编程语言 | `"java"` |
| `evidence` | ✅ 必填 | 证据描述 | 说明在代码中发现的问题，**字数≥20** |
| `impact` | ✅ 必填 | 影响描述 | 说明漏洞的危害和潜在风险，**字数≥20** |
| `remediation` | ✅ 必填 | 修复方案 | 必须包含具体代码示例、API名称或配置参数，**字数≥30** |
| `safeValidation` | ✅ 必填 | 安全验证建议 | 说明如何验证修复效果 |
| `codeSnippet` | ✅ 必填 | 问题代码片段 | 实际的问题代码 |
| `status` | ✅ 必填 | 状态 | `"误报"` 或 `"有效"` |

### 2. 内容质量要求

#### 问题描述（evidence）

**要求**：
- ✅ 必须说明漏洞成因
- ✅ 必须说明攻击方式
- ✅ 必须说明潜在风险
- ✅ 字数≥20

**合格示例**：
```
在 src/controllers/AuthController.java:45 发现认证绕过漏洞。攻击者可以通过构造特殊的 HTTP 请求绕过身份验证，未授权访问系统资源。该漏洞源于认证逻辑中存在条件判断缺陷，允许特定参数跳过认证检查。
```

**不合格示例**：
```
发现认证问题。
```
（原因：未说明漏洞成因、攻击方式、潜在风险，字数不足）

#### 修复方案（remediation）

**要求**：
- ✅ 必须提供具体修复方法
- ✅ 必须说明修复原理
- ✅ 必须包含代码示例或 API 建议
- ✅ 字数≥30

**合格示例**：
```
使用 Spring Security 的 @PreAuthorize 注解进行权限验证。在认证流程中添加完整的身份验证逻辑，确保所有敏感操作前都执行认证检查。示例代码：@PreAuthorize("isAuthenticated()") @GetMapping("/admin") public String adminPage() { ... }。修复原理：通过 Spring Security 的声明式安全机制，在方法调用前自动执行权限验证，防止未授权访问。
```

**不合格示例**：
```
修复认证问题。
```
（原因：未提供具体修复方法、修复原理、代码示例，字数不足）

#### 严重等级（severity）

**要求**：
- ✅ 必须符合漏洞实际危害
- ✅ 必须考虑上下文和可利用性
- ✅ 必须考虑影响范围

**严重等级评估标准**：

| 等级 | 条件 | 示例 |
|------|------|------|
| **高危（high）** | 可直接导致系统被入侵、数据泄露 | 认证绕过、SQL 注入、命令注入 |
| **中危（medium）** | 可导致权限提升或信息泄露，但需要特定条件 | CSRF、会话固定、信息泄露 |
| **低危（low）** | 存在安全隐患但影响较小 | 日志记录不完整、弱密码策略 |

### 3. 国标映射格式

**要求**：
- ✅ 必须使用正确的国标映射格式
- ✅ Java/C/C++/C# 必须使用双映射
- ✅ 其他语言使用单映射
- ✅ 分隔符使用中文分号 `；`

**正确格式示例**：
```
Java: GB/T34944-6.3.1.2 身份鉴别被绕过；GB/T39412-6.3.1.2 身份鉴别被绕过
C/C++: GB/T34943-6.2.3.3 命令注入；GB/T39412-6.1.1.6 命令行注入
C#: GB/T34946-6.2.3.3 命令注入；GB/T39412-6.1.1.6 命令行注入
Python: GB/T39412-6.1.1.1 输入验证不足
JavaScript: GB/T39412-6.1.1.5 HTTP头注入
```

**错误格式示例**：
```
GB/T34944-6.3.1.2 身份鉴别被绕过, GB/T39412-6.3.1.2 身份鉴别被绕过
```
（原因：使用了英文逗号而非中文分号）

### 4. 行号准确性

**要求**：
- ✅ 必须使用 Grep 工具验证行号
- ✅ 行号必须对应实际代码行（非注释行、非空行）
- ✅ 禁止凭记忆填写行号
- ✅ 禁止根据函数名推断行号

**验证步骤**：
```bash
# 1. 使用 Grep 精确查找
Grep pattern="问题代码关键字" path="目标文件" output_mode="content" -n

# 2. 确认行号对应实际代码
# 检查该行是否为代码行（非注释行、非空行）
```

## 🚫 禁止行为清单

**LLM 审计过程中禁止以下行为**：

### 1. 独立性违规

- ❌ 查看快速扫描结果（违反独立性原则）
- ❌ 根据快速扫描结果调整审计策略
- ❌ 重复检测快速扫描已覆盖的高风险函数调用

### 2. 审计范围违规

- ❌ 仅审计"关键代码文件"
- ❌ 跳过配置文件、工具类、实体类
- ❌ 仅审计 Controller/Service 层

### 3. 行号验证违规

- ❌ 凭记忆填写行号
- ❌ 根据函数名推断行号
- ❌ 不验证行号对应实际代码
- ❌ 使用注释行号

### 4. 内容质量违规

- ❌ 问题描述字数<20
- ❌ 修复方案字数<30
- ❌ 修复方案不包含代码示例或 API 建议
- ❌ 严重等级不符合实际危害

### 5. 国标映射违规

- ❌ 使用错误的国标映射格式
- ❌ Java/C/C++/C# 不使用双映射
- ❌ 使用英文逗号而非中文分号

### 6. 状态判定违规

- ❌ 状态字段填写"待判定"
- ❌ 不分析安全防护措施
- ❌ 不检查是否仅导入无调用
- ❌ 不检查是否测试代码

## 📊 修复方案示例库

### 合格修复方案示例

#### 示例 1：SQL 注入修复

```
使用 PreparedStatement 参数化查询替代字符串拼接。示例代码：
PreparedStatement pstmt = conn.prepareStatement("SELECT * FROM users WHERE username = ? AND password = ?");
pstmt.setString(1, username);
pstmt.setString(2, password);
ResultSet rs = pstmt.executeQuery();
修复原理：将用户输入作为参数而非 SQL 语句的一部分，数据库驱动会自动进行转义处理，防止 SQL 注入攻击。
```

#### 示例 2：认证绕过修复

```
使用 Spring Security 的 @PreAuthorize 注解进行权限验证。示例代码：
@PreAuthorize("isAuthenticated()")
@GetMapping("/admin")
public String adminPage() {
    return "admin";
}
修复原理：通过 Spring Security 的声明式安全机制，在方法调用前自动执行权限验证，防止未授权访问。同时建议在认证流程中添加完整的身份验证逻辑，确保所有敏感操作前都执行认证检查。
```

#### 示例 3：命令注入修复

```
使用 ProcessBuilder 替代 Runtime.exec。示例代码：
ProcessBuilder pb = new ProcessBuilder("grep", "-i", searchTerm, filePath);
pb.redirectErrorStream(true);
Process process = pb.start();
修复原理：ProcessBuilder 将命令和参数分开传递，避免 shell 解释特殊字符，防止命令注入攻击。同时建议对用户输入进行白名单验证，只允许预期的字符和格式。
```

#### 示例 4：XSS 修复

```
使用 Thymeleaf 模板引擎的自动转义功能。示例代码：
<p th:text="${userInput}"></p>
修复原理：Thymeleaf 的 th:text 属性会自动对输出进行 HTML 转义，将特殊字符转换为 HTML 实体，防止 XSS 攻击。同时建议对所有用户输入进行验证和清理，使用 OWASP Java Encoder 库进行上下文相关的编码。
```

## 🔧 常见问题解决方案

### 问题 1：如何判断漏洞严重程度？

**解决方法**：
1. 分析漏洞是否可直接利用
2. 分析漏洞的影响范围
3. 分析漏洞的利用条件
4. 参考 CVSS 评分标准

### 问题 2：如何确定国标映射？

**解决方法**：
1. 根据编程语言选择专用标准（Java → GB/T 34944）
2. 根据漏洞类型选择对应规则编号
3. 添加通用基线 GB/T 39412-2020
4. 使用中文分号分隔

### 问题 3：如何验证行号准确性？

**解决方法**：
1. 使用 Grep 工具精确搜索
2. 确认行号对应实际代码行
3. 禁止凭记忆填写
4. 禁止根据函数名推断

### 问题 4：如何避免误报？

**解决方法**：
1. 分析代码上下文（前后 10 行）
2. 检查是否有安全防护措施
3. 检查是否仅导入无调用
4. 检查是否测试代码

## 📝 质量检查总结

**LLM 审计质量取决于以下因素**：

1. **独立性**：完全独立于快速扫描，不查看快速扫描结果
2. **全面性**：审计全部代码文件，不自我设限
3. **准确性**：行号精确对应实际代码，不凭记忆填写
4. **完整性**：必填字段完整，内容质量达标
5. **规范性**：国标映射格式正确，严重等级合理

**违反后果**：
- 行号错误会导致验证失败，记录为幻觉
- 内容质量不达标会影响审计报告可信度
- 国标映射错误会影响合规性审查
- 严重等级不合理会影响漏洞修复优先级
